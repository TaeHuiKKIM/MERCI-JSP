[1. 프로젝트 개요 및 설계 철학 (40초)]
안녕하십니까, 쇼핑몰 프로젝트 'MERCI' 발표를 맡은 [이름]입니다.
저희 프로젝트의 핵심 목표는 단순한 기능 구현을 넘어, **'실제 서비스 가능한 수준의 사용자 편의성(UX)과 시스템 유지보수성'**을 확보하는 것이었습니다.

이를 위해 전체적인 아키텍처는 JSP와 Servlet을 기반으로 하되, **모듈화(Modularization)** 설계를 적극 도입했습니다. 모든 페이지에 공통적으로 들어가는 상단 헤더(`header.jsp`)와 하단 푸터(`footer.jsp`)를 별도 파일로 분리하고 `jsp:include` 액션 태그로 관리함으로써, 메뉴나 레이아웃 변경 시 유지보수 비용을 획기적으로 줄였습니다.

---

[2. 사용자 편의성을 고려한 디테일 구현 (1분 20초)]
사용자가 웹사이트를 이용하면서 느낄 수 있는 사소한 불편함을 제거하는 데 집중했습니다.

첫째, **'엔터키 로그인'** 기능입니다. 로그인 폼에서 비밀번호 입력 후 마우스로 로그인 버튼을 누르지 않아도, JavaScript의 `onkeypress` 이벤트를 감지하여 엔터키(keyCode 13) 입력 시 자동으로 폼(`loginForm`)이 제출되도록 구현했습니다.

둘째, **'조회수(Frequency) 기반 알고리즘'**입니다. `ClothDao`에 `updateFreq` 메소드를 구현하여 상품 상세 페이지(`catalogdetail.jsp`) 진입 시마다 해당 상품의 `freq` 컬럼 값을 1씩 증가시킵니다. 이를 통해 단순히 등록순이 아니라, 고객들의 관심도가 높은 상품을 '인기순'으로 정렬하여 노출할 수 있는 기반 데이터를 마련했습니다.

셋째, **'비동기 중복 체크'**입니다. 회원가입 시 페이지 리로딩 없이 `fetch` API(AJAX)를 사용해 아이디 중복 여부를 실시간으로 피드백 주어 가입 이탈률을 낮췄습니다.

---

[3. 외부 API 활용 및 결제 시스템 (1분 30초)]
실무에 가까운 기능을 구현하기 위해 다양한 외부 API를 매시업(Mashup)했습니다.

[카카오 로그인 & 주소 API]
접근성을 높이기 위해 **'카카오 로그인 SDK'**를 연동하여 원클릭 로그인을 구현했고, 배송지 입력 단계에서는 **'다음(Daum) 우편번호 API'**를 활용해 도로명 주소를 정확하고 간편하게 입력받을 수 있도록 했습니다.

[카카오페이 결제 연동]
가장 중요한 결제 시스템은 **'카카오페이(KakaoPay) API'** 테스트 모드를 연동했습니다. 
주문 폼(`order_form.jsp`)에서 결제 요청을 보내면, 결제 준비(Ready) -> 승인(Approve) 단계를 거쳐 실제 결제 프로세스와 동일한 흐름을 타게 됩니다. 이 과정에서 결제 고유 번호(TID)와 주문 정보를 세션(Session)에 저장하여 보안을 유지했습니다.

---

[4. 관리자 기능 및 데이터베이스 설계 (1분)]
관리자(Admin) 관점에서의 효율성도 놓치지 않았습니다.
관리자 페이지는 일반 사용자와 철저히 분리(`session.getAttribute`로 권한 체크)되었으며, 시각적인 데이터 분석을 위해 **'Chart.js'** 라이브러리를 활용해 매출 및 카테고리 현황을 그래프로 시각화했습니다.
또한 상품 등록 시 **'MultipartRequest'**를 이용해 다중 이미지 업로드 처리를 구현했으며, `User` 테이블에 최근 `findQ`(질문), `findA`(답변) 컬럼을 추가하는 마이그레이션을 진행하여, 이메일 인증을 대체하는 자체적인 보안 검증 로직을 완성했습니다.

[통합 예상 질문 리스트 (50선)]

### [A. 기획 및 아키텍처]
1. Q. 왜 Spring 프레임워크가 아닌 JSP/Servlet을 사용했나요?
   A. 웹의 근간이 되는 서블릿 컨테이너의 동작 원리와 HTTP 요청/응답 흐름(Request/Response Lifecycle)을 바닥부터 이해하기 위해 순수 JSP 모델을 채택했습니다.

2. Q. MVC 패턴을 적용했나요?
   A. 완전한 프레임워크 수준은 아니지만, JSP(View) - Java Bean(Model) - DAO/Process JSP(Controller 역할)로 역할을 분리하는 Model 1 아키텍처를 변형하여 적용했습니다. 로직과 뷰의 결합도를 낮추려 노력했습니다.

3. Q. Header와 Footer를 분리한 이유는 무엇인가요? (유지보수)
   A. 코드 중복(DRY 원칙 위배)을 막기 위해서입니다. 메뉴 항목 하나를 수정할 때 모든 페이지를 열지 않고 `header.jsp` 하나만 수정하면 사이트 전체에 반영되도록 설계했습니다.

4. Q. 세션(Session)은 주로 어디에 사용했나요?
   A. 로그인 상태 유지(`userId`, `userName`), 관리자 권한 체크, 장바구니 임시 저장, 그리고 결제 진행 중의 주문 정보 유지에 사용했습니다.

5. Q. 쿠키(Cookie) 대신 세션을 선택한 이유는?
   A. 보안 때문입니다. 사용자 정보 같은 중요 데이터는 클라이언트(브라우저)가 아닌 서버 메모리에 저장되는 세션이 더 안전하다고 판단했습니다.

### [B. 프론트엔드 (UI/UX)]
6. Q. 메인 페이지 슬라이더는 라이브러리인가요?
   A. 아닙니다. `window.innerWidth`로 화면 크기를 계산하여 `transform: translateX` 속성을 조절하는 바닐라 자바스크립트(Vanilla JS) 로직을 직접 구현했습니다.

7. Q. 모바일 반응형 처리는 어떻게 했나요?
   A. CSS Media Query(`@media`)를 사용하여 화면 너비 768px 이하일 때 그리드 레이아웃을 1단으로 변경하고, 폰트 크기를 조절했습니다.

8. Q. 로그인 시 엔터키 기능 구현 원리는?
   A. `input` 태그의 `onkeypress` 이벤트에서 키 코드 13번(Enter)이 감지되면 자바스크립트의 `form.submit()` 함수를 호출하도록 했습니다.

9. Q. AJAX 비동기 통신은 어디에 쓰였나요?
   A. 회원가입 시 아이디 중복 체크, 장바구니 수량 변경, 찜하기(Wishlist) 기능 등 페이지 전체 리로딩 없이 즉각적인 반응이 필요한 곳에 사용했습니다.

10. Q. 제이쿼리(jQuery)를 사용한 이유는?
    A. DOM 조작(Selector 활용)과 이벤트 핸들링 코드를 간결하게 작성하고, AJAX 호출 시 크로스 브라우징 문제를 쉽게 해결하기 위해 사용했습니다.

### [C. 데이터베이스 (DB)]
11. Q. DB 연결 방식은 무엇인가요?
    A. DBCP(Database Connection Pool) 패턴을 사용했습니다. 미리 생성된 Connection 객체를 빌려 쓰고 반납하는 방식으로, 서버 부하를 줄이고 응답 속도를 높였습니다.

12. Q. `freq` (조회수) 카운팅 방식은?
    A. 상품 상세 페이지(`catalogdetail.jsp`)가 로드될 때 `ClothDao.updateFreq(id)`가 호출되어 해당 ID의 `freq` 컬럼을 `UPDATE` 쿼리로 +1 증가시킵니다.

13. Q. 비밀번호 찾기를 위해 DB를 어떻게 수정했나요?
    A. 기존 테이블 구조로는 본인 확인이 어려워 `ALTER TABLE user ADD COLUMN findQ ...` 쿼리를 실행해 질문과 답변 컬럼을 추가하고, 이 세 가지 정보(ID, 질문, 답변)가 일치해야 비밀번호를 재설정할 수 있게 했습니다.

14. Q. 상품 이미지 저장은 DB에 직접 하나요?
    A. 아닙니다. 이미지는 서버의 `uploadfile` 디렉토리에 파일 형태로 저장하고, DB에는 파일명(String)만 저장합니다. DB 용량 절약과 성능을 위한 일반적인 패턴입니다.

15. Q. 트랜잭션 처리는 되어 있나요?
    A. 결제 후 주문 테이블(`Orders`)과 주문 상세 테이블(`OrderDetail`)에 동시에 데이터를 넣을 때, 하나의 논리적 작업으로 묶여야 하므로 `conn.setAutoCommit(false)`를 이용해 트랜잭션 처리를 고려했습니다.

16. Q. 인덱스(Index) 튜닝을 고려했나요?
    A. 현재는 데이터 양이 적어 PK(Primary Key) 인덱스만 활용하고 있지만, 추후 상품 검색(`WHERE title LIKE...`) 성능 향상을 위해 검색 컬럼에 인덱스 추가를 고려할 수 있습니다.

### [D. 백엔드 로직 (Java/DAO)]
17. Q. DAO(Data Access Object) 패턴을 쓴 이유는?
    A. 비즈니스 로직과 DB 접근 로직을 분리하기 위해서입니다. JSP에서는 SQL을 몰라도 DAO의 메소드만 호출하면 데이터를 가져올 수 있어 코드가 깔끔해집니다.

18. Q. DTO(Data Transfer Object) 혹은 VO(Value Object)는 무엇인가요?
    A. `User` 클래스나 `Cloth` 클래스처럼 데이터를 담아서 계층 간(DB -> DAO -> JSP)에 전달하는 객체입니다.

19. Q. `multipart/form-data` 처리는 어떻게 했나요? (이미지 업로드)
    A. `cos.jar` 라이브러리의 `MultipartRequest` 클래스를 사용했습니다. 이를 통해 파일 업로드와 폼 파라미터 수신을 동시에 처리했습니다.

20. Q. 검색 기능(Search) 구현 방식은?
    A. SQL의 `LIKE` 연산자(`%keyword%`)를 사용하여 제목이나 설명에 키워드가 포함된 상품을 조회하도록 `selectLike` 메소드를 구현했습니다.

### [E. 외부 API 연동]
21. Q. 카카오 로그인은 OAuth2 방식인가요?
    A. 네, 개념적으로 유사합니다. Kakao SDK를 통해 인증 코드(Auth Code)를 받고 토큰을 발급받아 사용자 프로필(ID, 닉네임)을 요청하는 흐름입니다.

22. Q. 카카오 결제 시 결제 검증은 어떻게 하나요?
    A. 결제 승인 후 리턴받는 `tid`(결제 고유 번호)와 결제 금액이 요청 시점의 데이터와 일치하는지 서버 세션 데이터와 대조하여 검증합니다.

23. Q. 도로명 주소 API 데이터는 어떻게 DB에 넣나요?
    A. 팝업에서 선택한 주소 문자열(Road Address)을 자바스크립트가 받아서 폼의 `hidden` input에 넣고, submit 시 DB의 `address` 컬럼에 텍스트로 저장합니다.

### [F. 보안]
24. Q. 비밀번호는 암호화했나요?
    A. 현재는 학습 목적으로 평문(Plain Text) 저장했지만, 실무라면 SHA-256이나 BCrypt 해시 알고리즘을 적용해야 함을 알고 있습니다.

25. Q. SQL 인젝션 방어는?
    A. `PreparedStatement`를 사용했습니다. 파라미터 바인딩 방식(`?`)을 사용하여 입력값이 쿼리 문법으로 해석되는 것을 원천 차단했습니다.

26. Q. 관리자 페이지 접근 제어는?
    A. `admin/header.jsp` 등 상단에서 세션의 `userId`가 'admin'인지 체크하고, 아니면 메인으로 강제 리다이렉트(`response.sendRedirect`) 시키는 로직을 넣었습니다.

27. Q. XSS(교차 사이트 스크립트) 방어는?
    A. 게시판 글쓰기 시 `<script>` 태그 입력을 막기 위해 `replaceAll("<", "&lt;")` 처리를 하거나 JSTL `c:out` 태그를 사용하여 스크립트 실행을 방지했습니다.

### [G. 관리자 기능]
28. Q. 관리자 페이지의 차트는 무엇을 썼나요?
    A. **Chart.js**라는 자바스크립트 라이브러리를 CDN으로 불러와 사용했습니다. Canvas 태그 위에 데이터를 그려줍니다.

29. Q. 상품 수정 시 이미지는 어떻게 처리하나요?
    A. 새로운 이미지를 업로드하면 기존 파일을 덮어쓰거나 새 파일명으로 DB를 업데이트하고, 업로드하지 않으면 기존 이미지 파일명을 유지하는 분기 처리를 했습니다.

30. Q. 회원 강제 탈퇴 기능도 있나요?
    A. 네, 관리자 모드에서 `UserDao.deleteById`를 호출하여 악성 회원을 삭제할 수 있으며, `CASCADE` 제약 조건을 통해 해당 회원의 글이나 주문 내역 처리도 고려했습니다.

### [H. 트러블슈팅 & 기타]
31. Q. 한글 깨짐 문제는 없었나요?
    A. `request.setCharacterEncoding("UTF-8")`을 모든 JSP 상단이나 필터에 적용하여 POST 요청 시 한글 파라미터가 깨지는 것을 해결했습니다.

32. Q. 가장 어려웠던 에러는?
    A. DB 컬럼명 불일치 에러(`Column 'xxx' not found`)였습니다. Java 코드의 변수명(camelCase)과 DB 컬럼명(snake_case)이 달라서 매핑이 안 되던 것을, DAO의 SQL 문과 `rs.get` 부분을 일일이 대조하여 수정했습니다.

33. Q. 동시 접속자가 많으면 어떻게 되나요?
    A. 현재 구조(Model 1)는 트래픽이 몰리면 JSP 처리 부하가 커질 수 있습니다. 추후 Spring MVC로 전환하고 로드 밸런싱을 도입해야 해결될 문제입니다.

34. Q. 깃(Git) 협업 시 충돌(Conflict)은 없었나요?
    A. 같은 JSP 파일을 동시에 수정할 때 발생했습니다. 이를 해결하기 위해 기능별로 작업 파일을 명확히 나누고, 커밋 전에 항상 `pull`을 먼저 하는 습관을 들였습니다.

35. Q. 프로젝트 진행 중 의견 충돌은?
    A. DB 설계 단계에서 테이블 정규화 수준에 대해 의견이 달랐으나, 성능보다는 개발 편의성을 위해 적절히 반정규화하는 방향으로 합의했습니다.

36. Q. 페이지 이동 시 `forward`와 `redirect`의 차이는?
    A. `forward`는 요청 정보(`request`)를 유지하며 서버 내부에서 이동하고, `redirect`는 브라우저에게 새 URL로 재요청하라고 시키는 것이라 데이터가 초기화됩니다. 데이터 전달이 필요할 땐 `forward`를 썼습니다.

37. Q. JSTL(JSP Standard Tag Library)을 쓴 이유는?
    A. JSP 페이지 내에 지저분한 자바 코드(`<% ... %>`)를 줄이고, 태그 형태(`<c:if>`, `<c:forEach>`)로 로직을 구현하여 가독성을 높이기 위해서입니다.

38. Q. 향후 추가하고 싶은 기능은?
    A. 상품 리뷰에 사진 첨부 기능, 그리고 사용자 취향을 분석한 상품 추천 알고리즘을 도입해보고 싶습니다.

39. Q. 왜 MyBatis를 안 썼나요?
    A. SQL과 자바 코드를 분리해주는 장점은 알지만, 이번 프로젝트에서는 JDBC의 기본 동작(Connection, Statement, ResultSet)을 직접 코딩하며 DB 연동의 기초를 다지기 위해 날것의 JDBC를 사용했습니다.

40. Q. 이미지 로딩 속도 최적화는?
    A. 썸네일용 이미지를 별도로 생성하진 않았지만, CSS로 표시 크기를 제한했습니다. 추후에는 업로드 시 이미지 리사이징 라이브러리를 도입하고 싶습니다.

41. Q. 장바구니 데이터는 비회원도 유지되나요?
    A. 현재는 로그인 기반 세션에 저장하므로 회원만 가능합니다. 비회원 장바구니를 구현하려면 쿠키나 로컬 스토리지(Local Storage)를 활용해야 합니다.

42. Q. 주문 번호 생성 규칙은?
    A. 단순히 DB의 Auto Increment ID를 쓰기도 하지만, 날짜+난수 조합(예: 20231214-A1B2)으로 생성하여 보안과 유일성을 확보하려 노력했습니다.

43. Q. 페이징(Pagination) 처리는 했나요?
    A. 상품 목록이나 게시판에 데이터가 많아질 것을 대비해, SQL의 `LIMIT ?, ?` 구문과 페이지 번호 계산 로직을 사용하여 페이징을 구현했습니다.

44. Q. RESTful API 인가요?
    A. 엄밀한 REST API는 아니지만, URL 패턴이나 AJAX 요청 시 HTTP 메소드(GET/POST)를 구분하여 리소스 중심의 설계를 지향했습니다.

45. Q. 서버 배포 경험은?
    A. 로컬(localhost) 환경뿐만 아니라 AWS EC2나 호스팅 서버의 Tomcat에 WAR 파일을 배포하여 외부에서도 접속 가능하게 테스트했습니다.

46. Q. CSS 프레임워크(Bootstrap 등) 사용 안 한 이유는?
    A. 우리 쇼핑몰만의 독특한 감성(UI)을 표현하기 위해 기존 프레임워크의 정형화된 스타일을 피하고, 직접 CSS를 작성하여 커스터마이징했습니다.

47. Q. 자바스크립트의 `let`, `const`, `var` 구분은?
    A. ES6 문법을 준수하여 변수 재할당이 필요한 경우 `let`, 상수는 `const`를 사용하여 스코프 오염을 방지했습니다. `var`는 사용을 지양했습니다.

48. Q. 깃허브(GitHub) 관리는 어떻게?
    A. 기능별 브랜치 전략(Feature Branch)까지는 아니더라도, 커밋 메시지 컨벤션(feat, fix 등)을 지키려 노력하며 히스토리를 관리했습니다.

49. Q. 이 프로젝트의 가장 큰 아쉬움은?
    A. 테스트 코드(JUnit)를 작성하지 못해 수동 테스트에 의존한 점입니다. 리팩토링 시 사이드 이펙트를 잡기 어려웠습니다.

50. Q. 이 프로젝트를 한 문장으로 정의한다면?
    A. "기본에 충실하면서도 사용자 경험(UX)이라는 디테일을 놓치지 않은, 나의 첫 번째 풀스택 커머스 프로젝트"입니다.
